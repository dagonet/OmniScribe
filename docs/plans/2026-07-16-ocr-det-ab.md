# OCR Detection-Model A/B — Measurement & Decision (Sprint 13, 2026-07-16)

**Decision: keep `en_PP-OCRv3_det_mobile` as the default detector.** Ship a new
opt-in `ocr_det_lang` override (rapidocr `LangDet`: `en` / `ch` / `multi`)
instead of changing the default. No variant met the pre-committed materiality
bar. This closes two Phase 6 backlog items with data.

## Background

The default "EN det" resolves to `en_PP-OCRv3_det_mobile` — a PP-OCRv3-arch
model (filed under the v4 section of rapidocr's `default_models.yaml`). Two open
questions: (1) does a different detector yield "free quality" on the eval set;
(2) can the `det_lang=CH` auto-override for server/PP-OCRv5 variants be removed.

rapidocr resolves `Det.lang_type` through a **`LangDet`** enum with only three
members — `en`, `ch`, `multi`. The engine previously passed `LangRec` values for
det, which reach `en`/`ch` by value-overlap but can never select `multi` (no
`multi` in `LangRec`). Phase 0 added `ocr_det_lang` (an override applied as
`LangDet(value)`) so the multilingual detector is reachable through the real
pipeline — the prerequisite for measuring it.

## Method

- **Harness:** `scripts/eval_ocr.py` (OCR-only; full sampling → UI filter →
  frequency filter → dedup → fuzzy scoring vs ground truth). GPU
  (CUDAExecutionProvider), local gitignored eval fixtures.
- **Det configs** (via `OMNI_OCR_*` env; rec + `ocr_language` held at each
  sample's GT language, so only the detector varies):
  - `base` — no override → `en_PP-OCRv3_det_mobile`
  - `multi` — `OMNI_OCR_DET_LANG=multi` → `multi_PP-OCRv3_det_mobile`
  - `ch-v5-server` — `OMNI_OCR_DET_MODEL_TYPE=server` + `OMNI_OCR_DET_OCR_VERSION=PP-OCRv5` → `ch_PP-OCRv5_det_server`
- **Samples:** latin-script s2 (DE photo), s3 (DE video), s4 (EN photo),
  s5 (DE photo) — all recall-saturated at 1.0; plus s6 (EN video, the one
  recall-headroom sample, baseline 0.607).
- **Metrics:** recall, precision, mean-match-similarity, and funnel
  `raw_bboxes` (a **cost proxy** — more raw detections ⇒ more rec forward
  passes; not a measured latency/VRAM number).
- **Decision gate (pre-committed):** regression guard = any recall below
  baseline disqualifies; win = aggregate similarity **or** precision ≥ +0.02
  vs base, no recall regression. For s6, a recall lift ≥ +0.05 reopens the
  decision.

## Core pass — recall-saturated samples (s2–s5)

| config | s2 P/sim | s3 P/sim | s4 P/sim | s5 P/sim | **agg P** | **agg sim** | raw range |
|---|---|---|---|---|---|---|---|
| base | 0.250 / 0.979 | 0.667 / 0.973 | 0.983 / 0.995 | 0.824 / 0.960 | **0.681** | **0.9768** | 36–75 |
| multi | 0.250 / 0.983 | 0.654 / 0.976 | 0.983 / 0.996 | 0.831 / 0.972 | 0.680 | 0.9818 (+0.0050) | 186–333 (3–5×) |
| ch-v5-server | 0.250 / 0.974 | 0.667 / 0.967 | 0.966 / 0.996 | 0.803 / 0.983 | 0.672 | 0.9800 (+0.0032) | 54–166 |

Recall = **1.000 for every config on every sample.** Best similarity lift is
`multi` at **+0.0050 aggregate** — 4× under the +0.02 bar — and it costs 3–5×
raw detections. Precision improves nowhere. No config wins.

## Headroom pass — s6 (baseline recall 0.607)

| config | recall | precision | sim | raw | post-freq |
|---|---|---|---|---|---|
| base | 0.607 | 0.634 | 0.970 | 176 | 71 |
| multi | 0.643 (+0.036) | 0.707 (+0.073) | 0.962 | 507 (2.9×) | 58 |
| ch-v5-server | 0.607 | 0.544 | 0.972 | 442 | 68 |

`base` and `ch-v5` stay flat at 0.607 — confirming the s6 gap is **dominantly
`filter_by_frequency` dropping the persistent title banner** (root-caused
prior), not detection. `multi` recovered a modest +0.036 recall / +0.073
precision — a **secondary** detector-completeness effect — but that is below the
+0.05 reopen bar, on one small-n sample, at 2.9× raw detections.

## Validity

- **The knob really swapped the detector.** `raw_bboxes` diverges wildly across
  configs (s2: 36 / 186 / 166; s6: 176 / 507 / 442) — not a silent no-op. This
  matters because rapidocr's OmegaConf param merge silently absorbs wrong
  `Det.*` keys (the Sprint 9.4 bug class); the raw-count divergence is positive
  proof the override took effect.
- **Detection is language-agnostic.** recall stays 1.0 even with the Chinese
  detector (`ch-v5-server`) on German/English text — box detection locates text
  regions regardless of script. This directly supports the registry-gap note:
  the server/v5 CH-force costs **no** latin recall.

## Decision & rationale

**Keep `en_PP-OCRv3_det_mobile` default.** `multi` is the only candidate with
any edge, and it is marginal on easy content (+0.005 sim) and sub-threshold on
hard content (+0.036 recall), always at 3–5× raw detections. Making it the
default would tax the common already-1.0 case with a large detection→rec cost
for a marginal gain — a poor default.

**Ship `ocr_det_lang` as an opt-in override.** It is additive (None default =
byte-identical behavior), completes the `ocr_det_*` knob family, and makes this
measurement reproducible. `multi` is a reasonable opt-in for hard / low-recall
content where the extra detection completeness earns its cost — not a default.

Default unchanged ⇒ **no test baseline move** (`Det.lang_type=EN` default
assertions stay valid).

## Phase 6 backlog — both items closed

- **v3-EN-det retirement** — measured. No detector beat en-v3 on the eval set
  under the materiality bar; default retained; `multi` exposed via `ocr_det_lang`.
- **RapidOCR det-model registry gap** — **not buildable in-repo.** rapidocr
  3.8.1 ships only `ch_*` det models for `server`/PP-OCRv5 (verified across all
  backends). The `det_lang=CH` auto-override is the correct workaround for an
  upstream model-distribution limitation; removing it needs an externally
  trained `en_*`/`multi_*` server/v5 det model (out of scope). Reclassified from
  buildable gap to documented upstream constraint.

(Aside: s2's 0.25 precision is identical across all three detectors — it is the
separate det-independent "OCR noise on text-heavy backgrounds" Phase 6 item, not
addressable by a detector swap.)
